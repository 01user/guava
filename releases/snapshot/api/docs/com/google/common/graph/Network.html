Interface Network<N,E>

    @Beta
public interface Network<N,E>
extends Graph<N>
    A network consisting of a set of nodes of type N and a set of edges of type E. That is, a subtype of Graph that represents edges as explicit first-class objects. Users that are not interested in edges as first-class objects should use a Graph instead.

    Users that wish to modify a Network must work with its subinterface, MutableNetwork.

    This interface permits, but does not enforce, any of the following variations of graphs:

    • directed and undirected edges
    • nodes and edges with attributes (for example, weighted edges)
    • nodes and edges of different types (for example, bipartite or multimodal graphs)
    • parallel edges (multiple edges which connect a single set of nodes)

    Extensions or implementations of this interface may enforce or disallow any or all of these variations.

    Definitions:

    • edge and node are incident to each other if the set of edge's endpoints includes node.
    • nodeA and nodeB are mutually adjacent if both are incident to a common edge.
      Similarly, edge1 and edge2 are mutually adjacent if both are incident to a common node.
    • Elements are connected if they are either incident or adjacent.
    • edge is an incoming edge of a node if it can be traversed (in the direction, if any, of edge) from a node adjacent to node.
    • edge is an outgoing edge of node if it can be traversed (in the direction, if any, of edge) from node to reach a node adjacent to node.
      • Note: undirected edges are both incoming and outgoing edges of a node, while directed edges are either incoming or outgoing edges of node (and not both, unless the edge is a self-loop).
        Thus, in the following example edge1 is an incoming edge of nodeB and an outgoing edge of nodeA, while edge2 is both an incoming and an outgoing edge of both node3 and node4: 
        
         directedGraph.addEdge(edge1, nodeA, nodeB);
         undirectedGraph.addEdge(edge2, node3, node4);
    • A node pred is a predecessor of node if it is incident to an incoming edge of node (and is not itself node unless edge is a self-loop).
    • A node succ is a successor of node if it is incident to an outgoing edge of node (and is not itself node unless edge is a self-loop).
    • Directed edges only:
      • node is a source of edge if edge is an outgoing edge of node.
      • node is a target of edge if edge is an incoming edge of node.

    General notes:

    • Nodes/edges must be useable as Map keys:
      • They must be unique in a graph: nodes nodeA and nodeB are considered different if and only if nodeA.equals(nodeB) == false, and the same for edges.
      • If you would otherwise have duplicate edges (e.g. weighted edges represented by a Double), you can instead wrap the edges in a custom class that defers to Object for its equals() and hashCode() implementations.
      • If graph elements have mutable state:
        • the mutable state must not be reflected in the equals/hashCode methods (this is discussed in the Map documentation in detail)
        • don't construct multiple elements that are equal to each other and expect them to be interchangeable. In particular, when adding such elements to a graph, you should create them once and store the reference if you will need to refer to those elements more than once during creation (rather than passing new MyMutableNode(id) to each add*() call).

      Generally speaking, your design may be more robust if you use immutable nodes/edges and store mutable per-element state in a separate data structure (e.g. an element-to-state map).
    • There are no Node or Edge classes built in. So you can have a Graph<Integer, String> or a Graph<Author,Publication> or a Graph<Webpage,Link>.
    • This framework supports multiple mechanisms for storing the topology of a graph, including:
      • the Graph implementation stores the topology (for example, by storing a Map<N, E> that maps nodes onto their incident edges); this implies that the nodes and edges are just keys, and can be shared among graphs
      • the nodes store the topology (for example, by storing a List<E> of incident edges); this (usually) implies that nodes are graph-specific
      • a separate data repository (for example, a database) stores the topology

    Notes on accessors:

    • Accessors which return collections may return views of the Graph. Modifications to the graph which affect a view (e.g. calling addNode(n) or removeNode(n) while iterating through nodes()) are not supported and may result in ConcurrentModificationException.
    • Accessors which return collections will return empty collections if their inputs are valid but no elements satisfy the request (for example: adjacentNodes(node) will return an empty collection if node has no adjacent nodes).
    • Accessors will throw IllegalArgumentException if passed a node/edge that is not in the graph.
    • Accessors take Object parameters rather than N/E generic type specifiers to match the pattern set by the Java Collections Framework.

    Notes for implementors:

    • Implementations have numerous options for internal representations: matrices, adjacency lists, adjacency maps, etc.
    • For accessors that return a Set, there are several options for the set behavior, including:
      1. Set is an immutable copy (e.g. ImmutableSet): attempts to modify the set in any way will throw an exception, and modifications to the graph will not be reflected in the set.
      2. Set is an unmodifiable view (e.g. Collections.unmodifiableSet()): attempts to modify the set in any way will throw an exception, and modifications to the graph will be reflected in the set.
      3. Set is a mutable copy: it may be modified, but modifications to the graph will not be reflected in the set, and vice versa.
      4. Set is a modifiable view: it may be modified, and modifications to the graph will be reflected in the set (but modifications to the set will not be reflected in the graph).
      5. Set exposes the internal data directly: it may be modified, and modifications to the graph will be reflected in the set, and vice versa.
      Note that (1) and (2) are generally preferred. (5) is generally a hazardous design choice and should be avoided, because keeping the internal data structures consistent can be tricky.
    • Prefer extending AbstractGraph over implementing Graph directly. This will ensure that the implementations of equals(Object) and hashCode() are mutually consistent, and consistent across implementations.
    • Multimaps are not sufficient internal data structures for Graph implementations that support isolated nodes (nodes that have no incident edges), due to their restriction that a key either maps to at least one value, or is not present in the Multimap.

    Examples of use:

    • Is node in the graph? 
      
   graph.nodes().contains(node)
    • Traversing an undirected graph node-wise: 
      
   // Visit nodes reachable from node.
   void depthFirstTraverse(N node) {
     if (!isVisited(node)) {
       visit(node);
       for (N successor : graph.successors(node)) {
         depthFirstTraverse(successor);
       }
     }
   }
    • Traversing a directed graph edge-wise: 
      
   // Update the shortest-path distances of the successors to node
   // in a directed graph (inner loop of Dijkstra's algorithm):
   void updateDistances(N node) {
     nodeDistance = distances.get(node);
     for (E outEdge : graph.outEdges(node)) {
       N target = graph.target(outEdge);
       double targetDistance = nodeDistance + outEdge.getWeight();
       if (targetDistance < distances.get(target)) {
         distances.put(target, targetDistance);
       }
     }
   }

      • equals

        boolean equals(@Nullable
             Object object)
        Returns true iff object is a graph that has the same node/edge relationships as those in this graph.

        Thus, two graphs A and B are equal if all of the following are true:

        • A and B have the same node set
        • A and B have the same edge set
        • A and B have the same incidence relationships, e.g., for each node/edge in A and in B its incident edge/node set in A is the same as its incident edge/node set in B.
          Thus, every edge in A and B connect the same nodes in the same direction (if any).

        Graph properties are not respected by this method. For example, two graphs may be considered equal even if one allows parallel edges and the other doesn't. Additionally, the order in which edges or nodes are added to the graph, and the order in which they are iterated over, are irrelevant.

        A reference implementation of this is provided by Graphs.equal(Network, Network).

      • hashCode

        int hashCode()
        Returns the hash code for this graph. The hash code of a graph is defined as the hash code of a map from each of the graph's nodes to their incident edges.

        A reference implementation of this is provided by Graphs.hashCode(Graph).
