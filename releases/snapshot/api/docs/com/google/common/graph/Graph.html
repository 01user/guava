com.google.common.graph

Interface Graph<N>

  • Type Parameters:
    N - Node parameter type
    All Known Subinterfaces:
    Hypergraph<N,E>, MutableGraph<N>, MutableNetwork<N,E>, Network<N,E>
    All Known Implementing Classes:
    AbstractGraph, AbstractNetwork, ImmutableGraph, ImmutableNetwork

    @Beta
public interface Graph<N>
    A graph consisting of a set of nodes of type N and a set of (implicit) edges. Users that want edges to be first-class objects should use the subtype Network instead.

    For convenience, we may use the term 'graph' to refer to any subtype of Graph, including Network and Hypergraph.

    Users that wish to modify a Graph must work with its subinterface, MutableNetwork.

    This interface permits, but does not enforce, any of the following variations of graphs:

    • directed and undirected edges
    • hyperedges (edges which are incident to arbitrary sets of nodes)
    • nodes and edges with attributes (for example, weighted edges)
    • nodes and edges of different types (for example, bipartite or multimodal graphs)
    • internal representations as matrices, adjacency lists, adjacency maps, etc.

    Extensions or implementations of this interface may enforce or disallow any or all of these variations.

    Definitions:

    • node1 and node2 are mutually adjacent (or connected in graph if an edge has been added between them: 
      
       graph.addEdge(node1, node2);  // after this returns, node1 and node2 are adjacent
      In this example, if graph is directed, then:
      • node1 is a predecessor of {code node2} in graph
      • node2 is a successor of node1 in graph
      • node1 has an (implicit) outgoing edge to node2 in graph
      • node2 has an (implicit) incoming edge from node1 in graph
      If graph is undirected, then:
      • node1 and node2 are mutually predecessors and successors in graph
      • node1 has an (implicit) edge in graph that is both outgoing to to node2 and incoming from node2, and vice versa.
    • A self-loop is an edge that connects a node to itself.

    General notes:

    • Nodes must be useable as Map keys:
      • They must be unique in a graph: nodes node1 and node2 are considered different if and only if node1.equals(node2) == false.
      • If graph elements have mutable state:
        • the mutable state must not be reflected in the equals/hashCode methods (this is discussed in the Map documentation in detail)
        • don't construct multiple elements that are equal to each other and expect them to be interchangeable. In particular, when adding such elements to a graph, you should create them once and store the reference if you will need to refer to those elements more than once during creation (rather than passing new MyMutableNode(id) to each add*() call).

      Generally speaking, your design may be more robust if you use immutable nodes and store mutable per-element state in a separate data structure (e.g. an element-to-state map).
    • There are no Node classes built in. So you can have a Graph<Integer> or a Graph<Author> or a Graph<Webpage>.
    • This framework supports multiple mechanisms for storing the topology of a graph, including:
      • the Graph implementation stores the topology (for example, by storing a Map<N, N> that maps nodes onto their adjacent nodes); this implies that the nodes are just keys, and can be shared among graphs
      • the nodes store the topology (for example, by storing a List<E> of adjacent nodes); this (usually) implies that nodes are graph-specific
      • a separate data repository (for example, a database) stores the topology

    Notes on accessors:

    • Accessors which return collections may return views of the Graph. Modifications to the graph which affect a view (e.g. calling addNode(n) or removeNode(n) while iterating through nodes()) are not supported and may result in ConcurrentModificationException.
    • Accessors which return collections will return empty collections if their inputs are valid but no elements satisfy the request (for example: adjacentNodes(node) will return an empty collection if node has no adjacent nodes).
    • Accessors will throw IllegalArgumentException if passed an element that is not in the graph.
    • Accessors take Object parameters rather than generic type specifiers to match the pattern set by the Java Collections Framework.

    Notes for implementors:

    • For accessors that return a Set, there are several options for the set behavior, including:
      1. Set is an immutable copy (e.g. ImmutableSet): attempts to modify the set in any way will throw an exception, and modifications to the graph will not be reflected in the set.
      2. Set is an unmodifiable view (e.g. Collections.unmodifiableSet()): attempts to modify the set in any way will throw an exception, and modifications to the graph will be reflected in the set.
      3. Set is a mutable copy: it may be modified, but modifications to the graph will not be reflected in the set, and vice versa.
      4. Set is a modifiable view: it may be modified, and modifications to the graph will be reflected in the set (but modifications to the set will not be reflected in the graph).
      5. Set exposes the internal data directly: it may be modified, and modifications to the graph will be reflected in the set, and vice versa.
      Note that (1) and (2) are generally preferred. (5) is generally a hazardous design choice and should be avoided, because keeping the internal data structures consistent can be tricky.
    • Prefer extending AbstractGraph over implementing Graph directly. This will ensure consistent equals(Object) and hashCode() across implementations.
    • Multimaps are not sufficient internal data structures for Graph implementations that support isolated nodes (nodes that have no incident edges), due to their restriction that a key either maps to at least one value, or is not present in the Multimap.

    Examples of use:

    • Is node in the graph? 
      
   graph.nodes().contains(node)
    • Traversing an undirected graph node-wise: 
      
   // Visit nodes reachable from node.
   void depthFirstTraverse(N node) {
     for (N neighbor : graph.adjacentNodes(node)) {
       if (!isVisited(neighbor)) {
         visit(neighbor);
         depthFirstTraverse(neighbor);
       }
     }
   }

        boolean isDirected()
        Returns true if the edges in this graph have a direction associated with them.

        A directed edge is an outgoing edge of its source, and an incoming edge of its target. An undirected edge connects its incident nodes to each other, and is both an outgoing edge and incoming edge of each incident node.

      • equals

        boolean equals(@Nullable
             Object object)
        Returns true iff object is a graph that has the same node relationships as those in this graph.

        Thus, two graphs A and B are equal if all of the following are true:

        • A and B have the same node set
        • A and B have the same adjacency relationships, e.g., for each node the sets of successor and predecessor nodes are the same in both graphs.
          This implies that every edge in A and B connects the same nodes in the same direction (if any).

        Graph properties are not respected by this method. For example, two relations may be considered equal even if one allows self-loops and the other doesn't. Additionally, the order in which edges or nodes are added to the graph, and the order in which they are iterated over, are irrelevant.

        A reference implementation of this is provided by Graphs.equal(Graph, Graph).

      • hashCode

        int hashCode()
        Returns the hash code for this graph. The hash code of a graph is defined as the hash code of a map from each of the graph's nodes to their adjacent nodes.

        A reference implementation of this is provided by Graphs.hashCode(Graph).

        Note that by this definition, two graphs that are equal in every aspect except edge direction will have the same hash code (but can still be differentiated by equals(java.lang.Object).
