Interface Graph<N,E>

  • Type Parameters:
    N - Node parameter type
    E - Edge parameter type

    @Beta
public interface Graph<N,E>
    A graph consisting of a set of nodes of type N and a set of edges of type E.

    This interface permits, but does not enforce, any of the following variations of graphs:

    • directed and undirected edges
    • hyperedges (edges which are incident to arbitrary sets of nodes)
    • nodes and edges with attributes (for example, weighted edges)
    • nodes and edges of different types (for example, bipartite or multimodal graphs)
    • parallel edges (multiple edges which connect a single set of vertices)
    • internal representations as matrices, adjacency lists, adjacency maps, etc.

    Extensions or implementations of this interface may enforce or disallow any or all of these variations.

    Definitions:

    • edge and node are incident to each other if the set of edge's endpoints includes node.
    • node1 and node2 are mutually adjacent if both are incident to a common edge.
      Similarly, edge1 and edge2 are mutually adjacent if both are incident to a common node.
    • Elements are connected if they are either incident or adjacent.
    • edge is an incoming edge of a node if it can be traversed (in the direction, if any, of edge) from a node adjacent to node.
    • edge is an outgoing edge of node if it can be traversed (in the direction, if any, of edge) from node to reach a node adjacent to node.
      • Note: undirected edges are both incoming and outgoing edges of a node, while directed edges are either incoming or outgoing edges of node (and not both, unless the edge is a self-loop).
        Thus, in the following example edge1 is an incoming edge of node2 and an outgoing edge of node1, while edge2 is both an incoming and an outgoing edge of both node3 and node4: 
        
         directedGraph.addEdge(edge1, node1, node2);
         undirectedGraph.addEdge(edge2, node3, node4);
    • A node pred is a predecessor of node if it is incident to an incoming edge of node (and is not itself node unless edge is a self-loop).
    • A node succ is a successor of node if it is incident to an outgoing edge of node (and is not itself node unless edge is a self-loop).
    • Directed edges only:
      • node is a source of edge if edge is an outgoing edge of node.
      • node is a target of edge if edge is an incoming edge of node.

    General notes:

    • Nodes/Edges must be unique, just as Map keys must be. So, nodes node1 and node2 are considered different if and only if node1.equals(node2) == false, and the same for edges.
    • There are no Node or Edge classes built in. So you can have a Graph<Integer, String> or a Graph<Author,Publication> or a Graph<Webpage,Link>.
    • This framework supports multiple mechanisms for storing the topology of a graph, including:
      • the Graph implementation stores the topology (for example, by storing a Map<N, E> that maps nodes onto their incident edges); this implies that the nodes and edges are just keys, and can be shared among graphs
      • the nodes store the topology (for example, by storing a List<E> of incident edges); this (usually) implies that nodes are graph-specific or map
      • a separate data repository (for example, a database) stores the topology
    • Users that are not interested in edges as first-class objects can create an implementation of (or subinterface of, or class delegating to) Graph that only exposes node-related methods.
    • Multimaps can't be used as internal data structures for Graph implementations that support isolated nodes (nodes that have no incident edges), due to their restriction that a key either maps to at least one value, or is not present in the Multimap.

    Notes on accessors:

    • Accessors which return collections return unmodifiable views.
    • Accessors which return collections will return empty collections if their inputs are valid but no elements satisfy the request (for example: adjacentNodes(node) will return an empty collection if node has no adjacent nodes).
    • Accessors will throw IllegalArgumentException if passed a node/edge that is not in the graph.
    • Accessors take Object parameters rather than N/E generic type specifiers to match the pattern set by the Java Collections Framework.

    For accessors that return a Set, there are several options for the set behavior, including:

    1. Set is an immutable copy (e.g. ImmutableSet): attempts to modify the set in any way will throw an exception, and modifications to the graph will not be reflected in the set.
    2. Set is an unmodifiable view (e.g. Collections.unmodifiableSet()): attempts to modify the set in any way will throw an exception, and modifications to the graph will be reflected in the set.
    3. Set is a mutable copy: it may be modified, but modifications to the graph will not be reflected in the set, and vice versa.
    4. Set is a modifiable view: it may be modified, and modifications to the graph will be reflected in the set (but modifications to the set will not be reflected in the graph).
    5. Set exposes the internal data directly: it may be modified, and modifications to the graph will be reflected in the set, and vice versa.
    Note that (1) and (2) are generally preferred. (5) is generally a hazardous design choice and should be avoided, because keeping the internal data structures consistent can be tricky.

    Examples of common operations:

    • Is node in the graph?
      • graph.nodes().contains(n)
    • Traversing an undirected graph node-wise: 

      
   // visit nodes reachable from a given node n
   void depthFirstTraverse(N n) {
     for (N neighbor : graph.adjacentNodes(n)) {
       if (!isVisited(neighbor)) {
         visit(neighbor);
         depthFirstTraverse(neighbor);
       }
     }
   }
    • Traversing a directed graph edge-wise: 

      
   // visit nodes reachable from a given node n via each of n's outedges
   // TODO(user): create a utility method for getting the 'other' node associated with an edge
   // that will work with non-directed graphs
   void depthFirstTraverse(N n) {
     for (E outEdge : graph.outEdges(n)) {
       N successor = graph.target(outEdge);
       if (!isVisited(successor)) {
         visit(successor);
         depthFirstTraverse(successor);
       }
     }
   }

      • addEdge

        boolean addEdge(E edge,
              N node1,
              N node2)
        Adds edge to this graph, connecting node1 and node2 (optional operation).

        Edges must be unique, just as Map keys must be; they must also be non-null.

        If the graph is directed, node1 is edge’s source, node2 is edge’s target, and edge is an outgoing edge of node1 and an incoming edge of node2.

        If edge already connects node1 to node2 in this graph (in the specified order if order is significant, as for directed graphs, else in any order), then this method will have no effect and will return false.

        Behavior if node1 and node2 are not already elements of the graph is unspecified. Suggested behaviors include (a) silently adding node1 and node2 to the graph or (b) throwing IllegalArgumentException.

        Returns:
        true iff the graph was modified as a result of this call
        Throws:
        IllegalArgumentException - if edge already exists and connects nodes other than node1 and node2, or if the graph is not a multigraph and node1 is already connected to node2
        UnsupportedOperationException - if the add operation is not supported by this graph

      • equals

        @CheckReturnValue
boolean equals(@Nullable
                              Object object)
        Returns true iff object is the same type of graph (directed, undirected, hypergraph) as this graph, and the same node/edge relationships exist in both graphs.

        Thus, two graphs A and B are equal if all of the following are true:

        • A and B are of the same type (DirectedGraph, UndirectedGraph, Hypergraph)
        • A and B have the same node set
        • A and B have the same edge set
        • A and B have the same incidence relationships, e.g., for each node/edge in A and in B its incident edge/node set in A is the same as its incident edge/node set in B.
          Thus, even if a node has the same sets of adjacent nodes (neighbors) in both A and B, if the sets of edges by which node is connected to its adjacent nodes are not the same in both A and B, then A and B are not equal.

        Properties that are not respected by this method:

        • GraphConfig configurations. If two graphs are equal by the above criteria but have different configurations, they are still equal. (For example: two graphs may be considered equal even if one allows parallel edges and the other doesn't.)
        • Edge/node ordering. The order in which edges or nodes are added to the graph, and the order in which they are iterated over, are irrelevant.
